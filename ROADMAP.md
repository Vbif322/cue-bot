# Roadmap

Эволюция проекта из Telegram-бота для турниров в мультиплатформенный продукт.
Этапы упорядочены по приоритету; каждый — самостоятельный релиз.

## M1 — Веб-интерфейс для игроков — готово (июль 2026)

Браузерный доступ для игроков: отдельный SPA (`app/`) + общий бэкенд. Беспарольный вход
(код на почту / Telegram Login Widget) с отвязкой личности от Telegram (`user_identities`
с бэкфиллом — закрыло аудит S2-2). Монорепо на npm workspaces. Реализовано: просмотр
публичных турниров, регистрация, отчёт/подтверждение/спор по результатам матча, просмотр
сетки, профиль + статистика, центр уведомлений. Раздача по поддоменам: игрок — cuebot.ru,
админка — admin.cuebot.ru (Host-диспетчеризация в Node). Хвосты (мост из бота на сайт,
Mini App initData) — в TODO.md.

## Доработки турнирного ядра

Полировка основного турнирного флоу: фичи, заложенные в схеме/коде, но не доведённые
до конца. Не привязаны к конкретному релизу — делаются по мере необходимости.

Рекомендуемая очерёдность (не жёсткая, ориентир по приоритету):

1. **Фоновой планировщик** — сразу после M2. Фундамент и заодно фикс подвисших матчей.
2. **Поматчевое доведение** — поверх планировщика, когда многомесячный формат станет
   приоритетом.
3. **waitlist / DQ / мелкие хвосты** — по спросу, вклиниваются между делом.

Внимание к пересечению с **M2-7** (ввод результата по фреймам переписывает флоу
`reportResult`/`confirmResult`): авто-подтверждение по таймауту и самозапись игроков
трогают тот же confirm-флоу — строить их **поверх нового флоу M2-7**, не до него.

- **Фоновой планировщик джобов.** *(Когда: сразу после M2; авто-подтверждение —
  кандидат вытащить уже в M2, если матчи подвисают на реальных турнирах.)* Один
  самостоятельный сервис по образцу `dialogSessionSweep` (`src/index.ts`), на который
  вешаются отложенные/периодические задачи. Потребители:
  - **Авто-подтверждение результата по таймауту.** Если оппонент не реагирует на
    заявленный результат за отведённое время — автоматически подтверждать счёт заявителя
    и уведомлять. Логика была в удалённом `timeoutService`. *(Поверх confirm-флоу M2-7.)*
  - **Напоминания о матче.** За N часов до `matches.scheduledAt` — напоминание обоим
    игрокам. Тип `match_reminder` уже есть (`src/db/schema/notifications.ts`), но сейчас
    шлётся лишь единожды при назначении времени (`notifyMatchScheduled`).
  - **Дедлайны матчей.** По истечении срока на сыгранную встречу — техпоражение через
    `setTechnicalResult` (`src/services/matchService.ts`) и уведомление.
- **Поматчевое расписание (`per_match`) — доведение.** *(Когда: после планировщика,
  когда многомесячный формат в приоритете.)* Режим `schedule_mode = per_match`
  реализован сквозно для ручного назначения времени организатором (схема → сервис
  `setMatchSchedule` → бот `msch:*` → админ-SPA), но под сценарий «турнир идёт месяцами,
  матчи назначаются по мере доступности игроков» не хватает:
  - **Срок турнира.** Колонка `endDate` в `tournaments` (сейчас только `startDate`) +
    миграция, поле в визарде создания и админ-SPA, отображение диапазона дат.
    *(Тривиальный quick win — можно раньше.)*
  - **Самозапись/согласование времени игроками.** Handshake «игрок предложил время →
    соперник принял/отклонил» по образцу двухфазного подтверждения результата
    (`reportResult`/`confirmResult` в `src/services/matchService.ts`); сейчас время ставит
    только организатор/судья (гейт `canManageTournament`). *(Поверх confirm-флоу M2-7.)*
  - **Показ времени в игровом вебе.** `scheduledAt` есть в типах (`app/src/lib/types.ts`),
    но не рендерится в `Bracket`/`MatchModal` (`app/`) — вывести. *(Хвост M1, quick win —
    можно раньше.)*
  - Напоминания и дедлайны для этих матчей — через общий фоновой планировщик (см. выше).
- **Лист ожидания (waitlist).** *(Когда: по спросу / quick win.)* При достижении
  `maxParticipants` (`registerParticipant` в `src/services/tournamentService.ts`) вместо
  отказа ставить игрока в очередь; при освобождении места — авто-продвижение и
  уведомление. Переиспользовать уже объявленный тип уведомления `participant_limit_reached`
  (`src/db/schema/notifications.ts`).
- **Дисквалификация (DQ).** *(Когда: по спросу.)* Команда судьи/организатора, пишущая в
  существующую таблицу `disqualifications` (`src/db/schema/disqualifications.ts`), смена
  статуса участника на `disqualified` и техпоражение текущего матча через
  `setTechnicalResult` (`src/services/matchService.ts`). Уведомление — уже готовая
  `notifyDisqualification` (`src/services/notificationService.ts`).

## M2 — Виды бильярда и дисциплины ← текущий этап

Расширение турниров с одного снукера на несколько видов бильярда и разновидности внутри
каждого. Дёшево и с высокой отдачей: поле `discipline` в `tournaments` уже есть, значения
`pool` / `russian_billiards` лишь закомментированы. Раннее внедрение расширяет рынок ещё до
монетизации. Включает детальный счёт по фреймам и статистику (фундамент для рейтинга M4).
Разбивка на задачи — в TODO.md (раздел «M2»).

## M3 — Платёжная система с автоматическим распределением средств

- Приём оргвзносов / входных платежей за участие в турнире.
- Призовой фонд и автоматические выплаты победителям по итогам сетки.
- Автоматическое распределение средств (сплит): платформа / организатор / федерация /
  призовой фонд по настраиваемым правилам.
- Платёжный провайдер с поддержкой сплитов и выплат (ЮKassa / CloudPayments / Stripe
  Connect — выбрать под целевой рынок), вебхуки, идемпотентность, история транзакций,
  возвраты / споры.
- Финансовый учёт: `payments` / `transactions` / `payouts` + ledger; привязка
  платёжных/выплатных реквизитов к пользователю.
- Зависимость: вносит минимальную «платную» надстройку над турниром (оргвзнос задаёт
  организатор); полноценный веб для организаторов — M5.

## M4 — Рейтинговая система

Полноценный рейтинг игроков вместо нынешней статистики «сыграно/победы/поражения», которая
вычисляется на лету в `userStatsService.ts` и нигде не хранится.

- Рейтинг **отдельно по каждой дисциплине** (ELO или Glicko-2 — выбрать); хранение в новой
  таблице `player_ratings` (`userId` × дисциплина → значение + история изменений).
- Пересчёт при подтверждении результата матча (шов в `matchService.ts`), с учётом
  технических побед / walkover; устойчивость к коррекции результатов (каскад пересчёта).
- Таблицы лидеров: сервис + API-эндпоинт + команда бота; рейтинги по дисциплинам в профиле
  игрока (`me` / веб-профиль).
- Зависимость: опирается на дисциплины из M2.

## M5 — Организаторы в вебе + привязка Telegram

Создание и управление своими турнирами из браузера. Привязка Telegram-аккаунта к
веб-аккаунту (переиспользуем существующий код-/токен-мост). Ресурсно-ограниченные права.

## M6 — Мультиплатформа: VK / WeChat

Новые `provider` в `user_identities`; абстракция провайдера уведомлений поверх нынешней
Telegram-доставки (общий шов — `sendNotification`).

## M7 — Бот в групповом чате: анонсы и управление турниром

Выход бота в групповые чаты: сначала как дешёвый канал привлечения (бот живёт в
комьюнити-чатах и виден новым игрокам), затем — полноценная площадка для ведения турнира.
Сейчас бот де-факто работает только в личке; заготовленный guard `privateOnly()`
(`src/bot/guards.ts`) существует, но нигде не подключён.

- Привязка турнира к конкретному групповому чату: колонка `tournaments.groupChatId`, флоу
  привязки (например, детект `my_chat_member` или команда организатора внутри группы),
  авторизация — только организатор турнира.
- Read-only анонсы/уведомления: новый seam отправки в группу — `postToGroupChat(api, chatId,
  text, keyboard?)` по аналогии с `sendNotification` (`src/services/notificationService.ts`),
  но без записи в `notifications` (там `userId` `NOT NULL`, под группу не годится).
  Переиспользовать существующие чистые UI-билдеры: `buildTournamentMessage`
  (`src/bot/ui/tournamentUI.ts`) на анонс/расписание, `formatMatchCard`
  (`src/bot/ui/matchUI.ts`) на результаты матчей, `buildBracketView`
  (`src/bot/ui/bracketUI.ts`) на финальную сетку.
- Полноценное управление турниром прямо из группы: регистрация участников,
  отчёт/подтверждение результатов и просмотр сетки (inline-кнопки).
- Групповые права: админ чата как организатор/судья турнира; точечно подключить
  `privateOnly()` (`src/bot/guards.ts`) к мутирующим командам; per-tournament scope команд
  (`{ type: 'chat', chat_id: groupChatId }`) вместо одинакового `all_group_chats`/
  `all_private_chats` (`src/bot/commands.ts`), по аналогии с per-user scope в
  `roleCommands.ts`. Защита от чужих действий.
- Управление рассылкой по-прежнему в личке/вебе — группа только на приём.

Разбивка на задачи — в TODO.md (раздел «M7»), когда придёт очередь.

## M8 — Федерации и подписки

Федерации как сущность над турнирами (`federations` + `federation_members`, scoped-роли
owner / organizer / referee, `tournaments.federationId`). Глобальный `role='admin'` →
платформенный супер-админ. Подписки/планы на федерацию поверх платёжной инфраструктуры M3.
