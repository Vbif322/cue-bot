# Roadmap

Эволюция проекта из Telegram-бота для турниров в мультиплатформенный продукт.
Этапы упорядочены по приоритету; каждый — самостоятельный релиз.

## M1 — Веб-интерфейс для игроков ← текущий этап

Браузерный доступ для игроков: отдельный SPA + общий бэкенд. Нативные аккаунты
email/пароль с отвязкой личности от Telegram (`user_identities`). Монорепо на npm
workspaces. Возможности: просмотр публичных турниров, регистрация, отчёт/подтверждение/
спор по результатам матча, просмотр сетки, профиль + статистика, центр уведомлений.

## Доработки турнирного ядра

Полировка основного турнирного флоу: фичи, заложенные в схеме/коде, но не доведённые
до конца. Не привязаны к конкретному релизу — делаются по мере необходимости.

- **Лист ожидания (waitlist).** При достижении `maxParticipants`
  (`registerParticipant` в `src/services/tournamentService.ts`) вместо отказа ставить
  игрока в очередь; при освобождении места — авто-продвижение и уведомление.
  Переиспользовать уже объявленный тип уведомления `participant_limit_reached`
  (`src/db/schema/notifications.ts`).
- **Дисквалификация (DQ).** Команда судьи/организатора, пишущая в существующую таблицу
  `disqualifications` (`src/db/schema/disqualifications.ts`), смена статуса участника на
  `disqualified` и техпоражение текущего матча через `setTechnicalResult`
  (`src/services/matchService.ts`). Уведомление — уже готовая `notifyDisqualification`
  (`src/services/notificationService.ts`).
- **Авто-подтверждение результата по таймауту.** Если оппонент не реагирует на
  заявленный результат за отведённое время — автоматически подтверждать счёт заявителя
  и уведомлять. Логика была в удалённом `timeoutService`; при возврате — как фоновой
  джоб/планировщик в самостоятельном сервисе.

## M2 — Виды бильярда и дисциплины

Расширение турниров с одного снукера на несколько видов бильярда и разновидности внутри
каждого. Дёшево и с высокой отдачей: поле `discipline` в `tournaments` уже есть, значения
`pool` / `russian_billiards` лишь закомментированы. Раннее внедрение расширяет рынок ещё до
монетизации.

- Двухуровневая модель: **вид** (снукер / пул / русский бильярд; задел на карамболь) →
  **дисциплина** внутри вида:
  - снукер — 15 красных (классика), 10 красных, 6 красных;
  - пул — «восьмёрка» / «девятка» / «десятка»;
  - русский бильярд — свободная / комбинированная / динамичная пирамида.
- Рефактор поля `discipline` (`src/db/schema/tournaments.ts`) в пару «вид + дисциплина»,
  миграция Drizzle; разумные дефолты правил и `winScore` под каждую дисциплину.
- Выбор вида и дисциплины в визарде создания турнира
  (`src/bot/wizards/tournamentCreation/`) и в админ-SPA; все строки — на русском.
- Фундамент для рейтинга (M5): рейтинг считается отдельно по каждой дисциплине.

### Детальный счёт по фреймам и статистика

Хранение результата матча не агрегатом, а по фреймам — с очками обоих игроков в каждом
фрейме. Опирается на модель дисциплин (что считать фреймом и какие стат-поля показывать
зависит от вида) и закладывает фундамент под рейтинг M5.

- Новая таблица фреймов (`match_frames`: `matchId` → `frameNumber`, очки игрока 1 и игрока 2;
  победитель фрейма выводится). Агрегаты `player1Score`/`player2Score` в `matches`
  пересчитываются из фреймов.
- Захват **максимального брейка** (снукер): отдельное поле на матч/игрока поверх очков фреймов.
- Ввод результата по фреймам в боте взамен выбора финального счёта
  (`src/bot/ui/matchUI.ts`, `reportResult` в `src/services/matchService.ts`);
  двухфазное подтверждение сохраняется — оппонент подтверждает разбивку по фреймам.
- Тай-брейк «группа + плей-офф»: настоящая **разница очков** вместо выведенной `frameDiff`
  в `src/services/standingsService.ts`.
- Отображение разбивки по фреймам и макс. брейка в админ-SPA (вкладки матчей/standings).
- Детальное хранение очков — опционально и управляется дисциплиной (для дисциплин без
  «брейка» собирается только счёт фреймов).
- **Переменная длина матча по раундам/этапам**: `winScore` перестаёт быть единым на турнир —
  можно задавать большее число побед на поздних стадиях (напр. финал до 3, как растёт
  длина матчей по ходу профессионального снукера). Переопределение по раунду/стадии в
  схеме (`src/db/schema/tournaments.ts`), выбор в визарде создания
  (`src/bot/wizards/tournamentCreation/`) и в админ-SPA; валидация результата в
  `reportResult` сверяет счёт с `winScore` нужного раунда, а не турнирным.

## M3 — Бот в групповом чате: анонсы и уведомления

Выход бота в групповые чаты как дешёвый канал привлечения: бот живёт в комьюнити-чатах и
виден новым игрокам. Сейчас бот работает только в личке — везде стоит guard `privateOnly()`
(`src/bot/guards.ts`).

- Бот публикует в групповой чат: анонсы турниров, расписание, результаты матчей и финальную
  сетку (read-only, без мутаций состояния в группе).
- Точечное смягчение guard-ов / скоупов команд для групп; привязка идентификатора группового
  чата к турниру для адресной рассылки.
- Переиспользуем существующий шов уведомлений; управление по-прежнему в личке / вебе.

## M4 — Платёжная система с автоматическим распределением средств

- Приём оргвзносов / входных платежей за участие в турнире.
- Призовой фонд и автоматические выплаты победителям по итогам сетки.
- Автоматическое распределение средств (сплит): платформа / организатор / федерация /
  призовой фонд по настраиваемым правилам.
- Платёжный провайдер с поддержкой сплитов и выплат (ЮKassa / CloudPayments / Stripe
  Connect — выбрать под целевой рынок), вебхуки, идемпотентность, история транзакций,
  возвраты / споры.
- Финансовый учёт: `payments` / `transactions` / `payouts` + ledger; привязка
  платёжных/выплатных реквизитов к пользователю.
- Зависимость: вносит минимальную «платную» надстройку над турниром (оргвзнос задаёт
  организатор); полноценный веб для организаторов — M6.

## M5 — Рейтинговая система

Полноценный рейтинг игроков вместо нынешней статистики «сыграно/победы/поражения», которая
вычисляется на лету в `userStatsService.ts` и нигде не хранится.

- Рейтинг **отдельно по каждой дисциплине** (ELO или Glicko-2 — выбрать); хранение в новой
  таблице `player_ratings` (`userId` × дисциплина → значение + история изменений).
- Пересчёт при подтверждении результата матча (шов в `matchService.ts`), с учётом
  технических побед / walkover; устойчивость к коррекции результатов (каскад пересчёта).
- Таблицы лидеров: сервис + API-эндпоинт + команда бота; рейтинги по дисциплинам в профиле
  игрока (`me` / веб-профиль).
- Зависимость: опирается на дисциплины из M2.

## M6 — Организаторы в вебе + привязка Telegram

Создание и управление своими турнирами из браузера. Привязка Telegram-аккаунта к
веб-аккаунту (переиспользуем существующий код-/токен-мост). Ресурсно-ограниченные права.

## M7 — Мультиплатформа: VK / WeChat

Новые `provider` в `user_identities`; абстракция провайдера уведомлений поверх нынешней
Telegram-доставки (общий шов — `sendNotification`).

## M8 — Бот в групповом чате: привязка турнира и управление

Полноценное управление турниром прямо из группового чата поверх анонсов M3.

- Привязка турнира к конкретному групповому чату; регистрация участников, отчёт/подтверждение
  результатов и просмотр сетки прямо в группе (inline-кнопки).
- Групповые права: админ чата как организатор / судья турнира; защита от чужих действий.
- Зависимость: расширяет M3.

## M9 — Федерации и подписки

Федерации как сущность над турнирами (`federations` + `federation_members`, scoped-роли
owner / organizer / referee, `tournaments.federationId`). Глобальный `role='admin'` →
платформенный супер-админ. Подписки/планы на федерацию поверх платёжной инфраструктуры M4.
